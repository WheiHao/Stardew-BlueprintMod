# Planting Feature Plan

这份计划用于扩展 `Farm Blueprint`，让玩家分享的蓝图可以包含“需要种植作物或花盆作物才能完整复现”的信息。

目标不是一开始就做成全自动种植，而是先把功能拆成稳定、可渐进上线的几个阶段。

## 产品目标

- 让蓝图作者能够表达“这个设计还需要种什么”
- 让使用者在导入蓝图时清楚知道还缺哪些种植步骤
- 在不破坏当前放置逻辑的前提下，逐步增加辅助种植能力
- 区分普通摆件放置与作物种植，降低失败率和兼容风险

## 推荐实现路线

### Phase 1: 记录种植计划，不自动种植

目标：

- 蓝图可以保存“这里应该种什么”
- 预览和浏览器可以显示种植需求
- 放置蓝图后，玩家知道后续该自己补种什么

范围：

- 在蓝图数据结构中新增种植计划字段
- 保存蓝图时识别地上作物和花盆中的作物
- 读取蓝图时保留这些数据
- 浏览器中显示“种植需求”或“作物清单”
- 放置后提示“该蓝图还有待种植内容”

为什么先做这个：

- 风险最低
- 最符合分享场景
- 不需要一次性解决季节、地块、浇水、温室、花盆等全部规则

### Phase 2: 辅助种植

目标：

- 在条件满足时，帮助玩家自动种植
- 将“建筑/装饰放置”与“作物补种”分成两个明确动作

范围：

- 新增“辅助种植”入口
- 检查玩家是否有对应种子
- 检查目标格是否可种植
- 检查地上种植与花盆种植的合法性
- 检查季节/室内/温室限制
- 给出失败原因提示

建议交互：

- 先放置蓝图主体
- 再弹出“是否继续辅助种植”的确认
- 若条件不足，则显示缺少的种子或不合法的种植位

### Phase 3: 创造模式全自动种植

目标：

- 在创造模式中直接完成蓝图对应的种植状态

范围：

- 在创造模式下忽略普通材料消耗
- 允许直接生成已种下的作物
- 明确区分这是创造模式专属能力

注意：

- 这一阶段适合最后做
- 逻辑最复杂，最容易引入边界问题

## 建议的数据结构

推荐不要把作物完全混进现有 `BlueprintItem`，而是单独增加一组数据。

示例结构：

```json
{
  "Metadata": {
    "Name": "Spring Bee Garden",
    "Width": 12,
    "Height": 8
  },
  "Items": [],
  "PlantingPlans": [
    {
      "TileX": 3,
      "TileY": 4,
      "Mode": "Ground",
      "SeedItemId": "(O)472",
      "CropId": "BlueJazz",
      "DisplayName": "Blue Jazz",
      "RequiresWateredHoeDirt": true,
      "Season": "spring"
    },
    {
      "TileX": 8,
      "TileY": 2,
      "Mode": "IndoorPot",
      "SeedItemId": "(O)433",
      "CropId": "CoffeeBean",
      "DisplayName": "Coffee Bean",
      "RequiresPot": true
    }
  ]
}
```

建议字段：

- `TileX`
- `TileY`
- `Mode`：`Ground` / `IndoorPot`
- `SeedItemId`
- `CropId`
- `DisplayName`
- `RequiresPot`
- `RequiresWateredHoeDirt`
- `Season`

## 代码改动建议

### 第一批应改文件

- `ModEntry.cs`
- `BlueprintBrowserMenu.cs`

### 建议新增文件

- `PlantingPlan.cs`
- `PlantingRequirement.cs` 或类似命名的数据类文件

### Phase 1 的代码任务

1. 新增 `PlantingPlan` 数据类
2. 给 `BlueprintFile` 增加 `PlantingPlans`
3. 在保存蓝图时扫描：
   - 地上已种植的作物
   - 花盆中的作物
4. 在加载蓝图时读取种植计划
5. 在浏览器里显示种植清单
6. 在预览 HUD 或放置确认中提示该蓝图包含种植内容

### Phase 2 的代码任务

1. 新增“辅助种植”流程入口
2. 编写合法性检查：
   - 是否有花盆
   - 是否是可耕地
   - 是否有对应种子
   - 是否符合季节/位置要求
3. 编写失败提示文案
4. 把种植结果加入撤销或单独记录

### Phase 3 的代码任务

1. 为创造模式单独处理自动种植
2. 明确与普通模式分支
3. 完善撤销与边界检查

## UI 建议

为了更贴近“分享作品”的目标，建议在蓝图浏览器里增加一块种植信息区域。

推荐显示：

- 该蓝图包含多少个种植位
- 需要哪些种子
- 哪些是地上种植
- 哪些需要花盆
- 当前季节是否适合该蓝图

推荐文案示例：

- `该蓝图包含 24 个地面种植位`
- `该蓝图包含 8 个花盆种植位`
- `当前季节可能无法完整复现该蓝图中的全部作物`

## 风险点

- 作物与普通物件的保存规则不同
- 地上种植依赖锄地状态、浇水状态和季节
- 花盆种植依赖花盆对象本身是否存在
- 不同地图区域对种植合法性影响较大
- 自动种植如果和一键放置强绑定，失败体验会很差

因此建议始终坚持：

- 放置与种植拆开
- 先提示，再自动化
- 先支持“记录与展示”，再支持“自动执行”

## 推荐开发顺序

1. 先实现 `PlantingPlans` 数据结构
2. 实现蓝图保存时记录地上作物和花盆作物
3. 在浏览器中显示种植需求
4. 在放置后提示“还有待种植内容”
5. 再做“辅助种植”
6. 最后再考虑创造模式全自动

## 当前结论

最合适的实现方式是：

- 蓝图先记录“种植意图”
- 主体摆件先正常放置
- 种植作为后续独立步骤
- 在条件成熟后，再增加自动化能力

这样最稳，也最符合“分享好看的农场装修”这个产品方向。
